# Инструкции для coding-агентов

## Назначение

Репозиторий создаёт локальный игровой вертикальный срез «КОНТУР v0.3». Текущий
результат — один консольный эпизод «Кислотный фронт» на 15–20 минут, а не
Telegram-приложение, production-сервис или универсальный игровой движок.

Отвечай владельцу по-русски. Имена Java-типов и стабильные ID пиши по-английски.

## Источники истины

Перед планированием и изменением кода прочитай:

1. `ACID_FRONT_SPEC.md` — исполнимая механика и переходы эпизода.
2. `REQUIREMENTS.md` — функциональные и технические требования.
3. `KONTUR_V03_GAME_DESIGN.md` — продуктовая гипотеза и границы.
4. `IMPLEMENTATION_PLAN.md` — этапы и ворота.

Если документы расходятся, не угадывай и не исправляй механику ради тестов.
Назови точные пункты и запроси решение владельца.

## Разрешённый объём

Разрешено:

- Java 21 и Gradle Wrapper с Kotlin DSL;
- один локальный консольный модуль;
- неизменяемая доменная модель;
- явная реализация `AcidFrontEpisode`;
- ограниченный `EpisodeDirector` по нормативному графу сцен;
- seeded `2d6` и внедряемый `DiceRoller`;
- доктрина из `M`, `A`, `R`;
- JUnit 5, AssertJ и локальные отчёты проверки.

Запрещено до ворот P3:

- Spring Boot и DI-фреймворки;
- Telegram, HTTP, React и Mini App;
- PostgreSQL, Redis, Kafka и файловое сохранение;
- сеть, внешние API и LLM в runtime;
- авторизация, платежи и деплой;
- универсальный DSL, редактор сценариев или процедурный мир;
- подключение `research-prototype-relay17` как зависимости;
- второй эпизод и месячная кампания.

## Технологические решения

- Java 21 через Gradle toolchain.
- Gradle Wrapper 9.6.1, Kotlin DSL, один модуль.
- Корневой Java-пакет: `game.kontur.prototype`.
- Production-код только в `src/main/java`.
- Тесты: JUnit 5 и AssertJ.
- Не добавлять mocking-фреймворки.
- Не использовать текущее время, UUID, окружение или глобальное состояние в
  игровом ядре.
- Сценарий кодировать явно; обобщать только после второго настоящего эпизода.
- `d20` не добавлять: нормативная проверка v0.3 использует `2d6`.

## Архитектурные правила

- `WorldState`, `HiddenWorldState`, `EpisodeSessionState` и результаты переходов
  неизменяемы.
- `EpisodeDirector` выполняет роль цифрового мастера: выбирает ровно одну
  следующую авторскую сцену по стадии и состоянию. Он не импровизирует.
- `WorldState`, `HiddenWorldState` и публичное представление разделены.
- Публичный переход принимает состояние и `PlayerDecision`; выбор варианта и
  ввод доктрины представлены разными типами. Он возвращает новое состояние и
  упорядоченный список `GameEvent`.
- Состояние содержит местоположение и доступность каждой машины; `LOST`,
  `STRANDED` и `AWAY` не смешиваются с физическим повреждением.
- Скрытые факты, публичный briefing и причинный отчёт — разные представления.
- `DiceRoller` передаётся явно. Одинаковый seed и решения воспроизводимы.
- Чтение текущего шага и preview никогда не расходуют случайность.
- Логика не зависит от текста интерфейса.
- Неприменимый выбор отклоняется без частичного изменения состояния.
- Атмосферный и причинный текст строятся из одного журнала событий.
- Общий score и шкалу морали не добавлять.

## Рабочий процесс

Перед изменением:

- проверь рабочее дерево;
- прочитай связанные разделы документов;
- сформулируй проверяемый инвариант и ограниченный план.

После изменения:

- запусти узкие тесты;
- запусти `./gradlew test`;
- перед завершением запусти `./gradlew clean check`;
- просмотри сгенерированные отчёты, а не только exit code;
- перечисли выполненное, отклонения и оставшиеся ворота.

Не создавай remote, ветки, коммиты и PR без прямой просьбы владельца. Не изменяй
старый репозиторий Relay-17.

## Условия остановки

Остановись и запроси решение, если:

- для перехода не хватает правила в `ACID_FRONT_SPEC.md`;
- два документа дают разные эффекты одного `ChoiceId`;
- автоматический обход находит тупик или хотя бы один из семи профилей
  недостижим;
- директор выбирает ноль или несколько следующих сцен;
- единственный способ пройти тест — изменить игровой смысл;
- возникает желание начать инфраструктуру до человеческого теста.
