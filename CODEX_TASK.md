# Первая задача для Codex CLI

Статус: **COMPLETED / ARCHIVE**. Задача уже была выполнена при создании v0.3.
Не запускай её повторно. Текущая разрешённая задача находится в
`NARRATIVE_V04_CODEX_TASK.md`.

Ниже находится готовый текст задания. Его следует передать Codex после создания
пустого отдельного репозитория `kontur-game-prototype` и копирования в корень
всех файлов этого пакета.

---

Изучи все документы репозитория, особенно `AGENTS.md`, `ACID_FRONT_SPEC.md`,
`REQUIREMENTS.md` и `IMPLEMENTATION_PLAN.md`.

Цель: реализовать только этапы 1–4 из `IMPLEMENTATION_PLAN.md` — локальный
консольный вертикальный срез «Кислотный фронт». Не начинай Telegram, Spring,
HTTP, базу данных, сохранения, LLM-интеграцию или универсальный сценарный
движок. Не изменяй и не подключай репозиторий `research-prototype-relay17`.

Перед реализацией:

1. Проверь состояние рабочего дерева.
2. Составь короткий план по этапам: домен → эпизод → доктрина и риск → CLI →
   полный обход и отчёты.
3. Проверь спецификацию на блокирующие противоречия. Сообщай только те, без
   решения которых невозможно однозначно реализовать переход. Не превращай
   ревью в переписывание документов.
4. Если найдено блокирующее противоречие — остановись до ответа владельца.

Технологии:

- Java 21;
- Gradle Wrapper 9.6.1 с Kotlin DSL;
- один модуль;
- корневой пакет `game.kontur.prototype`;
- `application` plugin и запуск `./gradlew run`;
- JUnit 5 и AssertJ;
- никаких иных production-зависимостей без необходимости.

Реализуй:

1. Неизменяемые `WorldState`, `HiddenWorldState` и `EpisodeSessionState` с
   раздельными состояниями здоровья, модулей, местоположения и доступности машин.
2. Ограниченный `EpisodeDirector` в роли цифрового мастера: только нормативный
   граф, ровно одна следующая сцена, без LLM и импровизации.
3. `AcidFrontEpisode` со всеми переходами S1–S5 и эпилогом строго по
   `ACID_FRONT_SPEC.md`.
4. Все шесть перестановок доктрины `M/A/R` и пошаговую переоценку приказов.
5. Внедряемый `DiceRoller`, production-реализацию seeded `2d6` и фиксированную
   реализацию для тестов. `d20` не добавлять.
6. Единый упорядоченный журнал `GameEvent`, из которого строятся атмосферный и
   причинный отчёты.
7. Русский консольный интерфейс без знания доменной логики.
8. Автоматический обход всех доступных путей и отчёты из этапа 4 плана.

Публичный контракт движка зафиксирован в разделе 5 `REQUIREMENTS.md`. Не
кодируй доктрину как фиктивный `ChoiceId`: используй типизированные
`ChoiceDecision` и `DoctrineDecision`.

Обязательные проверки:

- `./gradlew test`;
- `./gradlew clean check`;
- ручной запуск `./gradlew run --args="--seed 1703"`;
- sample walkthrough:
  `DISCHARGE_BATTERIES → START_GREENHOUSE → M>A>R →
  POWER_RESCUE_BEACON → SEND_GREENHOUSE_CULTURES`;
- sample walkthrough завершается профилем `AGRARIAN_ALLIANCE`, населением 45,
  пищей `RATIONED_TWO_DAYS`, ограниченным реактором, отключённой мастерской,
  тремя сохранёнными машинами, союзом с `NODE_12` и исправным сенсором `STRIZH`;
- одинаковые seed и решения создают одинаковые события и финал;
- повторное чтение текущего шага не меняет состояние и не расходует кубики;
- все шесть доктрин соответствуют нормативной таблице;
- директор выбирает ровно одну сцену в каждом нетерминальном состоянии;
- контакт с `NODE_12` происходит ровно один раз в каждом прохождении;
- достижимы все семь итоговых профилей из спецификации;
- любой путь содержит ровно пять принятых игровых решений;
- нет тупиков, отрицательных ресурсов, неизвестных переходов и утечек скрытых
  фактов.

Не обновляй спецификацию, чтобы подогнать её под реализацию. Не создавай коммит,
ветку, remote или PR без отдельной команды владельца.

В финальном отчёте укажи:

- что реализовано;
- структуру production-кода;
- количество автоматически пройденных путей и итоговых профилей;
- матрицу шести доктрин;
- результаты обязательных команд;
- созданные отчёты;
- все отклонения от спецификации;
- открыты ли технические ворота P1 и что ещё должен проверить человек.
